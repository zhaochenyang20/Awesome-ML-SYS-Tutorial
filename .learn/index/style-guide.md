# 风格指南 (Style Guide)

> 本文件是 /learn agent 的核心风格约束，将被注入 LLM system prompt。所有 /learn-plan、/learn-write、/learn-review 的输出必须严格遵循以下规则。

---

## 1. 开篇结构（三段式）

每篇文章的开篇必须包含以下三个部分，按顺序排列：

### 1.1 个人动机/情感

- **必须**以第一人称坦诚交代写作动机，动机必须源自真实的工程需求、协作经历或知识焦虑，而非"今天来学习 X 概念"。
- **鼓励**包含个人感受、自嘲、或对学习过程的真实情绪表达。
- 典型模式：
  - 从工作需要出发："因为工作需要，最近终于得空能够再次深入去学习思考..."
  - 从协作中发现盲区："前几天和 XX 在研究 YY 的时候发现自己对 ZZ 知之甚少"
  - 从社区讨论引发焦虑："经常看到在技术讨论群里，大家聊的内容有许多我都毫不了解，让人产生了深深的技术焦虑"
  - 从具体对话/任务触发："XX 有次给我说过，'...'"
  - 表达文人相赏之情："越读起来，越有一种文人相赏，难逢知音的感觉"
  - 自嘲式引入："众所周知，我一直在 SGLang team 负责端茶倒水 + RLHF"

### 1.2 致谢

- 对于重要文章（非短篇笔记），**必须**包含致谢列表。
- 致谢格式：`姓名（学校/公司）`，多人用逗号分隔。
- **可以**加入轻松的排名说明，如"排名按照微信群的成员顺序"。
- 若引用了他人的工作或文档，也在此处致谢并附链接。
- 示例：

```
照理，感谢参与本文档讨论和撰写的所有朋友们：

zhuoran yin（CMU），changyi yang（CMU），ji li（蚂蚁），chengxi li（CMU）
```

### 1.3 文章路线图

- **必须**在正文前概述本文将讨论的内容，让读者知道全文走向。
- 典型模式：
  - 编号列表式："所以今天的文档尝试理解这些内容：1. ...; 2. ...; 3. ..."
  - 叙述式："这篇文章会 follow 以上三篇文章的思路，深入理解..."
  - 简要交代结构："本文首先分析...，接着...，最后...，欢迎大家批评指正"
- 若文章属于系列，**必须**标明系列编号和前序文章链接。

## 2. 概念拆解（定义 -> 数值例子 -> 代码）

每个核心概念的讲解必须遵循三层递进结构：

### 2.1 先下定义

- 用**加粗文本**给出概念的核心定义。
- 定义应当简洁、精确、自包含。
- 示例：**"在一个复杂的工作流程中，single controller 只有一个程序负责管理，而其他的子模块只负责执行。"**

### 2.2 再举数值化的具体例子

- **必须**用具体数字让抽象概念可感知。
- 数字应来自真实系统场景，而非虚构。
- 典型模式："假设这个参数的 size 是 `[1024, 1024]`，FSDP 的 TP size 是 4，而 SGLang 的 TP size 是 2。因此..."
- 又如："考虑大小为 XB 的模型，这三部分会有 12X 的显存占用。"

### 2.3 最后给代码

- 代码**必须**来自真实的生产系统（SGLang、verl、slime、Megatron 等），**禁止**使用纯教学示例。
- 代码块后**必须**有逐步分析，解释每行/每段的行为。
- 代码引用格式要求见第 4 节。

---

## 3. 知识框架构建

在进入代码分析之前，**必须**先用大篇幅建立概念性的理解框架。框架通常包含以下一种或多种模式：

### 3.1 分类对比

- 将相关概念按维度分类对比，譬如：single-controller vs multi-controller、co-locate vs hybrid engine、FULL_STATE_DICT vs LOCAL_STATE_DICT vs SHARDED_STATE_DICT。
- 对比**必须**从设计动机出发，而非仅列举表面差异。

### 3.2 设计权衡

- 明确点出每个设计选择背后的 trade-off。
- 典型句式："XX 的好处是...，然而...会导致..."、"当然，XX 和 YY 都可以...，不过速度自然会有所损失"。
- 鼓励给出"什么场景下用什么方案"的判断。

### 3.3 历史脉络/递进推演

- 按时间或复杂度递进，从简到繁地引出目标概念。
- 典型模式：DP -> DDP -> DeepSpeed Zero Stage 1/2/3 -> FSDP；Volta -> Hopper -> Blackwell。
- 每一级递进都要说明"为什么需要进化到下一级"。

### 3.4 工作流程分解

- 对于系统设计类内容，先给出完整的工作流程（如 PPO 的四步），再逐步深入每一步。
- 流程步骤用编号列表呈现，每步一句话概括。

## 4. 代码落地

### 4.1 来源要求

- 代码**必须**来自真实生产框架（SGLang、verl、slime、Megatron、OpenRLHF 等）。
- **禁止**使用自编的教学示例代替真实代码。如需简化说明，用伪代码并明确标注为伪代码（放在 `<details>` 折叠块中）。

### 4.2 引用格式（硬约束）

- 所有外部代码引用**必须**包含具体的 commit hash，格式为：
  `https://github.com/org/repo/blob/<commit_hash>/path/to/file.py#L123`
- **禁止**引用 main 分支上的行号（行号会随代码变更失效）。
- 若基于某个特定 commit 进行整篇分析，在文章开头标注："本文基于 commit `<hash>` 进行分析。"

### 4.3 代码分析方式

- 贴出关键代码片段后，**必须**逐步分析，解释每个关键步骤的行为。
- 分析粒度：对核心逻辑逐行分析，对辅助代码概括说明。
- 分析顺序：按代码执行顺序，从上到下。
- 若涉及多组件交互，画出调用链路或用文字描述完整链路（如 Server -> TokenizerManager -> Scheduler -> TpModelWorker -> ModelRunner）。

## 5. 过渡句式库

概念轨到代码轨的过渡**必须**使用明确的转折信号。以下是从已有文章中提取的过渡句式，按场景分类：

### 5.1 从概念框架进入代码分析

- "我们还是回到主线上"
- "有了这些认识，我们再来看..."
- "有了这个直观的理解，我们回顾下..."
- "好了，有了这些基础..."
- "理解了这些概念后，我们可以正式进入..."
- "总之，这篇文章的开篇花费了巨大的篇幅来简述...理解了这些概念后，我们可以正式进入..."

### 5.2 从一个子话题过渡到另一个

- "总之，..." 后接下一话题引入
- "接着，我们会剖析..."
- "最后，我们横向对比..."
- "这一部分是一些背景补充："
- "这里直接给结论了："
- "最后，XX 还带了一个好处，..."

### 5.3 引入设问/深入讨论

- "有一个问题非常值得分享：..."
- "这里思考一个小问题，..."
- "一个可能不太直观的点是——..."
- "注意到，..."
- "让我惊讶的是，..."

### 5.4 引入引用/参考

- "正如前文所提到的，..."
- "如同我之前提到的一样，..."
- "注：上面描述的...只是经验上的 practice，但实际上..."
- "坦诚说...（PS：...）"

### 5.5 总结/收尾

- "由此以来，..."
- "可见，..."
- "总的来说，..."
- "这就是一个新的故事了"
- "其中的酸甜苦辣，自不必多说"
- "知易行难..."

## 6. 深度层级判定规则

学习深度按"与作者工作的相近程度"分四级，写作时**必须**根据 topic 所在层级控制内容深度：

### 6.1 修改扩展级（modify-extend）

- **适用对象**：作者直接参与开发的系统（SGLang、verl、slime、SGLang-Omni）。
- **深度要求**：源码级分析，能理解设计权衡并提出改进方案。
- **内容特征**：完整代码走读、逐行分析、横向对比多种实现、给出个人判断。
- **代码量**：大量真实代码片段，附精确 commit hash 链接。

### 6.2 理解复现级（understand-reproduce）

- **适用对象**：依赖的基础设施（FSDP、NCCL、CUDA Graph、torch.distributed）。
- **深度要求**：理解设计思想和使用方式，能复现关键操作，但不深入内核实现。
- **内容特征**：概念讲解 + API 使用示例 + 关键原理分析，止步于"能正确使用并读懂输出"。
- **代码量**：适量代码示例，侧重 API 调用和配置。

### 6.3 建立直觉级（intuition）

- **适用对象**：相关的算法理论（PPO、RL Theory、量化理论、constraint decoding）。
- **深度要求**：建立直觉而非证明定理，止步于"能指导系统设计决策"。
- **内容特征**：保留"平凡到看似愚蠢的定义"，用直观解释替代严格证明，选择性深入与系统工作直接相关的部分。
- **代码量**：少量伪代码或数学公式，重在直观理解。

### 6.4 摘要提取级（summary）

- **适用对象**：读到的论文（Kimi K1.5、SWE-Bench 等）。
- **深度要求**：提取对系统工作有用的信息，浓墨重彩于与自身工作相关的部分。
- **内容特征**：全面但选择性的解读，技术报告式概述。
- **代码量**：极少或无代码，以文字概述为主。

### 6.5 深度一致性原则

```
自己开发的系统 (SGLang/verl/slime/SGLang-Omni) -> 源码级，能修改扩展
                |
依赖的基础设施 (FSDP/NCCL/CUDA Graph) -> 原理级，能正确使用
                |
相关的算法理论 (PPO/RL Theory/量化理论) -> 直觉级，能指导系统设计
                |
读到的论文 (Kimi K1.5/SWE-Bench) -> 摘要级，提取对系统工作有用的信息
```

若 topic 跨越多个层级，按**最相关部分**的层级为主，其余部分按各自层级控制深度。当作者明确调整了某个 topic 的深度偏好时，以最新指示为准。

## 7. 交叉引用触发条件

满足以下任一条件时，**必须**插入交叉引用：

### 7.1 前序依赖

- 当前内容依赖 repo 中已有文章的知识时，**必须**明确告知读者先读什么。
- 格式："实际上，我已经写过一篇前序文章：[文章标题](repo内路径)"
- 或者："这篇文章会 follow 以上三篇文章的思路"并列出链接。

### 7.2 同系列衔接

- 系列文章之间**必须**有明确的编号和衔接说明。
- 在文章开头或路线图中标明："本文是这系列的第 N 篇"。

### 7.3 跨主题引用

- 当讨论的概念在另一个主题的文章中有更详细的阐述时，**应当**引用。
- 格式："在 [文章标题](路径) 中已经介绍过了"、"具体的 XX 架构可以参考此图"。

### 7.4 引用格式

- repo 内引用：使用相对路径 Markdown 链接，如 `[文章标题](../../rlhf/sys-design/readme-1.md)`。
- 外部引用：使用完整 URL，优先引用知乎文章、官方文档、GitHub 代码（带 commit hash）。

## 8. 行文风格

### 8.1 谦逊但自信

- 坦诚承认不足："非常惭愧，从那之后，我也没有理解到二者的关系"、"恰好最近重新思考了下..."
- 同时敢于判断："我个人觉得..."、"让我惊讶的是..."、"希望以后我能有实力基于机器的物理属性直接分析得到这些结论"
- 展现学习者心态，而非全知全能。

### 8.2 适度幽默

- 允许在非核心位置插入轻松的表达：删除线吐槽（`~~便于绕开 sudo 用户直接 kill 僵尸进程~~`）、表情（"排名按照微信群的成员顺序"后加 emoji）。
- 幽默**不得**影响技术准确性和阅读流畅性。
- 幽默频率：每篇文章 1-3 处，点到为止。

### 8.3 主观判断

- **鼓励**在横向对比后给出个人判断："这里直接给结论了"、"知易行难...其中的心酸滋味，自然只有真的打磨过，才能体会"。
- 主观判断**必须**建立在充分分析之后，不能空洞表态。
- 判断用语："我个人觉得"、"在我看来"、"坦诚说"、"其中的酸甜苦辣，自不必多说"。

### 8.4 第一人称视角

- 全文使用第一人称"我"或"我们"（当涉及协作团队时）。
- 保持"学习笔记"而非"教科书"的基调。

### 8.5 自问自答

- 在关键概念转折处，使用设问句引导读者思考：
  - "有一个问题非常值得分享：为什么 XX 需要先 YY 再 ZZ，直接 AA 不行吗？"
  - "这里思考一个小问题，为什么不能...？简单来说，..."
- 设问后**必须**给出回答，回答可以简洁（"简单来说，太慢了"），也可以展开。

## 9. 格式惯例

### 9.1 加粗

- **关键概念的定义**必须加粗。
- **核心结论**必须加粗。
- 段落内的**重要术语**首次出现时加粗。
- 禁止滥用加粗——只有"如果读者只看加粗部分就能抓住要点"的内容才应加粗。

### 9.2 引用块

- 用引用块（`>`）标记以下内容：
  - 来自他人的重要观点或引用："推理引擎的 kernel fusion 和 training engine 差距不小..."
  - 重要的注意事项或旁注。
- 引用块**不得**用于大段的正文内容。

### 9.3 `<details>` 折叠块

- 用于可选的深入内容，保持主线清晰：
  - 伪代码（`<details><summary>伪代码</summary>...</details>`）
  - 补充证明或推导
  - 较长的代码片段（如果不影响主线理解）
- 折叠块的 `<summary>` **必须**简洁说明折叠内容。

### 9.4 代码块

- 所有代码使用 fenced code block，标注语言（python、cpp、bash 等）。
- 代码块前**必须**有一句话说明这段代码的来源和作用。
- 代码块后**必须**有分析文字。

### 9.5 图片和图表

- 架构图使用 HTML `<div>` 居中包裹：
  ```html
  <div style="text-align: center;">
    <img src="./pics/xxx.jpg" alt="描述" style="width:50%;">
  </div>
  ```
- Mermaid 图表用于流程描述（如调度流程、数据流转）。
- 图片**必须**有 alt 文本描述。

### 9.6 标题层级

- `#` 用于文章标题（全文唯一）。
- `##` 用于一级章节。
- `###` 用于二级章节。
- 标题应简洁，可包含中文术语。

### 9.7 列表

- 有序列表用于有顺序依赖的步骤（工作流程、执行顺序）。
- 无序列表用于并列的特性、注意事项。
- 列表项可以包含完整段落，但**每项首句必须概括要点**。

### 9.8 注释和补充

- 行内补充用圆括号："（PS：...）"、"（具体来说是 SPMD 的）"。
- 段落级补充用"注：..."或"PS：..."开头。

---

## 10. 语言约束

### 10.1 中文优先

- 所有写作以中文进行。
- 技术术语保留英文原文，不做生硬翻译（如 single-controller、rollout engine、handle tuple）。
- 中英文之间**不**加空格（遵循 repo 已有惯例）。

### 10.2 排除 [Pending Review] 文章

- README.md 和 README-cn.md 中标记为 `[Pending Review]` 的文章**绝对不能**作为风格参考或知识来源。

## 11. 三维度自检清单

/learn-write 输出内容后，**必须**按以下清单自检：

### 11.1 双轨检查

- [ ] 是否同时包含概念框架和代码分析？
- [ ] 概念框架是否在代码之前建立？
- [ ] 代码是否来自真实生产系统？
- [ ] 概念框架与代码分析之间是否有明确的过渡句？

### 11.2 叙事检查

- [ ] 开篇是否有个人动机？
- [ ] 是否包含致谢（对于重要文章）？
- [ ] 是否有文章路线图？
- [ ] 段落间是否有过渡句式？
- [ ] 是否在合适位置插入了交叉引用？
- [ ] 是否使用了设问句引导读者思考？

### 11.3 深度检查

- [ ] 当前 topic 在"距离梯度"中的位置是什么？
- [ ] 内容深度是否匹配该层级的要求？
- [ ] 是否有过深或过浅的部分？若有，是否有合理理由？
